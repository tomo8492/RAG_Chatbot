## 1. 現状(できていること)

完全ローカル(`offline / LAN_ONLY`)の RAG チャット。土台はすでに広い。

| 区分 | 実装済み |
|---|---|
| 検索 | ハイブリッド検索(密ベクトル + 語彙/BM25 を RRF 融合・**型番/条番号など完全一致に強い**)、見出し対応チャンク、ソース単位の多様化、追問のクエリ書き換え |
| 回答 | 厳格RAG(資料内のみ・「記載なし」と言う)、出典表示、ストリーミング/停止/再生成、思考の深さ調整 |
| 入力 | PDF/Word/Excel/PPT/テキスト、ファイル添付、画像コピペ/D&D(Vision)、増分インデックス |
| 出力 | Word/Excel/PPT/HTML/テキスト/Markdown 出力、一括要約(map-reduce・観点プリセット・裏実行) |
| 連携 | **OCR API**(`/api/ocr` — VBA等の外部から利用可)、Code エージェント(作業フォルダ内でファイル操作/コマンド実行・**差分承認制**) |
| 運用 | 複数会話/履歴、他PCアクセス(LAN)、簡易パスワード認証、ライト/ダーク |

→ 新テーマの多くは **この土台の再利用**で実現できる(下表「再利用」参照)。

## 2. 横断方針(全テーマ共通の鉄則)

設計・実装で必ず守る原則。テーマ個別設計より優先。

1. **オフライン厳守** — `offline / LAN_ONLY` を維持。社外送信なし。
2. **出典必須・厳格RAG** — 資料内のみで回答、無い時は「わからない」。
3. **数値・寸法・統計をLLMに生成させない** — 計算は道具(コード/集計/SQL)で確定し、
   LLMは**検索・下書き・翻訳・自然言語の説明**に使う(ハルシネーション防止)。
4. **品質・安全に関わる出力は人の確認(human-in-the-loop)** — 監査証跡を残す。
5. **実データで裏取りしてから実装** — 合成サンプルで確定せず、実物でチャンク/抽出を
   `evalkit` 等で実測してから着手(規定Phase1 §0 で得た教訓)。
6. **機微情報は非コミット** — 検証はローカルのみ。テストは匿名化したダミー最小例に限る。

## 4. 段階ロードマップ(推奨順)

2トラック(文書RAG=規定 / 工場ユースケース)を、**価値が早く出る順・依存の浅い順**で交互に。
カレンダー上の時期は体制次第のため、ここでは**フェーズ(相対順)**で示す。

```
フェーズ0  検索精度の土台         A1(規定の構造化チャンク+参照解決) + e5-base再索引 + 評価基盤
   │        └ 全文書テーマに波及(設備マニュアルB1にも効く)
フェーズ1  成功体験(RAGの強み)   B1 設備トラブル対応
フェーズ2  データ化              B2 帳票・検査記録の読み取り(1帳票種でPoC)
フェーズ3  製品・品質・図面       B3b 製品findability → B3c 品質(Code連携) → B3a 図面メタ
フェーズ4  規定の高度化(必要時)  A2 整合性チェック → A3 改訂履歴/版管理
```

### 各フェーズの狙いと依存

| フェーズ | 主作業 | 依存 | 規模(目安) |
|---|---|---|---|
| 0 | 規定A1の構造化・参照解決を実装、e5-base 再索引、`evalkit` で精度測定 | なし(既存の splitter/retrieval 拡張) | 中 |
| 1 | B1: トラブル履歴ローダ + エラーコードのメタ突合 + 「症状→原因→対策」出力 | フェーズ0の検索土台 | 中 |
| 2 | B2: 帳票テンプレ定義 + 構造化抽出 + **人手確認フロー** + 監査証跡 | 既存 `ocr.py`/`/api/ocr` | 中〜大 |
| 3 | B3b 構造化データの自然言語IF → B3c 品質は **Code エージェントで計算+説明** → B3a 図面タイトルブロックのメタ取込 | Code エージェント(既存) | 大(分割) |
| 4 | A2 same-source の参照グラフで矛盾検出 → A3 附則からの改訂履歴抽出・版フィルタ | A1 の条番号メタ・same-source 解決 | 中〜大 |

> **フェーズ0を先頭に置く理由**: 規定A1の「構造化チャンク+参照解決」は検索精度の底上げで、
> 規定だけでなく**設備マニュアル(B1)など長文PDF全般に効く**共通基盤になるため。

## 5. 共通基盤(横断で効く投資)

個別テーマと並行して整えると全体が楽になるもの。

- **評価ハーネス(`evalkit`)** — 各テーマの「実データで裏取り」をルーチン化(回帰検知)。
- **構造化チャンク/メタの拡充** — 見出し・ラベル・コードのメタ化は検索の質に直結(A1で着手)。
- **取り込みローダの拡張** — 履歴Excel(B1)・帳票テンプレ(B2)・台帳/PLM(B3)。共通I/Fで増やす。
- **人手確認 & 監査証跡の仕組み** — B2/B3c/品質系で再利用。低信頼度フラグ+原本リンク。
- **Code エージェント活用パターン** — 数値分析は「LLMに推定させず、コードで計算→人がレビュー→LLMが説明」を定型化(B3c)。

## 6. リスクと前提(横断)

| 事項 | 対応 |
|---|---|
| 実データのフォーマットの癖 | 着手前に実物で実測(方針5)。手書き帳票はOCR正答率を測ってから自動化範囲を決める |
| LLMの数値・図面の不正確さ | 方針3で生成させない。計算=道具、図面=メタ検索に限定 |
| 文書/データの鮮度 | 更新→再索引のオーナーシップを決める(運用ルール) |
| 機微情報 | 非コミット・ローカル検証(方針6) |
| 過剰期待 | 「自動化100%」を狙わず、人手確認込みの現実解で価値を出す |
