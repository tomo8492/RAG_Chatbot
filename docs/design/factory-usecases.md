# 設計/実現性:工場でのローカルLLM活用(3テーマ)

対象テーマ(選択):
1. 設備トラブル対応
2. 帳票・検査記録の読み取り
3. 製品データ・図面・品質データ

- ステータス: 設計/実現性検討(未実装)。**実データは後日受領 → 実物で裏取りしてから実装**(規定で得た教訓)。
- 前提: `offline / LAN_ONLY` 維持。安全・品質に直結するため **出典必須・人の確認(human-in-the-loop)** を全テーマ共通の鉄則とする。
- 再利用できる既存資産: ハイブリッド検索(`retrieval.rerank`/型番・コードの完全一致に強い)、見出し対応チャンク、出典付き回答・厳格RAG、Vision/OCR(`ocr.py`/`llm.vision_complete`/`/api/ocr`)、ファイル出力(`export.py`)、一括要約(`summarize.py`)、Codeエージェント(分析スクリプト実行)。

> **重要な原則**:LLMに「数値分析の答え」や「図面の正確な寸法」を**生成させない**。
> 計算は道具(コード/集計)にやらせ、LLMは**検索・下書き・翻訳・自然言語での説明**に使う。

---

## テーマ1: 設備トラブル対応  【実現性: 高】

**狙い**: 「設備Xでエラー E12」「○○が異音」から、マニュアル該当箇所＋**過去の対応履歴**を横断検索し、症状→原因→対策を出典付きで提示。ダウンタイム短縮・暗黙知の形式知化。

| 項目 | 内容 |
|---|---|
| データ源 | 設備マニュアル(PDF)、エラーコード表、**過去トラブル/対応履歴**(Excel・日報・保全記録) |
| アプローチ | RAG(既存の強み)。エラーコードは**完全一致語**なのでハイブリッド検索が効く。`error_code` をメタ化して厳密突合 |
| 新規要素 | トラブル履歴の**取り込み形式**(列定義: 日時/設備/症状/原因/対策/部品)。`loaders` に履歴ローダ追加。出力テンプレ「症状→原因→対策→出典」 |
| 既存再利用 | ハイブリッド検索・出典・厳格RAG・構造化チャンク |
| 実データ検証 | **履歴がどう記録されているか**(Excelの列・自由記述・写真添付)で取り込み方式が決まる → 受領後に確定 |
| 安全 | 最終対処の実施判断は人。危険作業はマニュアル原文へ誘導(LLMに手順を“創作”させない) |

**LLMが効く/効かない**: 効く=類似事例の検索・要約・手順の提示(出典付き)。効かない=設備状態の自律診断(センサ値の判断は人/別系)。

---

## テーマ2: 帳票・検査記録の読み取り  【実現性: 中(手書き精度が鍵)】

**狙い**: 手書き帳票・検査記録(紙/写真)を **Vision/OCR でデータ化**(CSV/DB)。転記作業の削減。

| 項目 | 内容 |
|---|---|
| データ源 | 手書き帳票、検査記録、ラベル(スキャン/写真) |
| アプローチ | Vision/OCR → **構造化抽出**(項目名→値をJSON/CSV)。`/api/ocr`・`vision_complete` が土台 |
| 新規要素 | **帳票テンプレ定義**(帳票種別ごとに抽出する項目)、抽出結果の**バリデーション**(数値範囲・必須)、**信頼度の低い項目の人手確認フロー**、監査証跡 |
| 既存再利用 | `ocr.py`(vlmエンジン)、`vision_complete`、`/api/ocr`(VBA等の外部連携済み) |
| 実データ検証 | **実帳票で OCR/抽出の正答率を測る**(手書きのクセ・かすれ・枠線)。テンプレごとに精度確認 → 受領後に確定 |
| 安全 | 検査記録は**品質記録** → 誤読は品質事故。**人の確認を必須化**し、原本画像を保管(追跡可能性) |

**LLMが効く/効かない**: 効く=活字・定型帳票の抽出、項目の構造化。効きにくい=崩れた手書き・低画質 → **人手確認前提**。100%自動化は狙わない。

## テーマ3: 製品データ・図面・品質データ  【実現性: 混在 — 切り分けが重要】

ここは**素朴なRAGが効くもの/効かないもの**が混ざるので分けて設計する。

### 3-a. 図面(CAD/PDF)  【実現性: 低〜中(メタ検索に限定)】
- 図面は**画像主体で本文テキストが乏しい**。LLMに幾何・寸法を“読ませる”のは不正確 → やらせない。
- 現実解: **タイトルブロックのメタ**(図番・品番・改訂・名称)を OCR または管理台帳/PLM から取り込み、**「この品番の図面はどれか」を探す findability 層**にする。図面そのものは人が開く。
- 実データ検証: タイトルブロックの体裁・図番の表記ゆれを実図面で確認。

### 3-b. 製品データ(仕様・BOM)  【実現性: 中〜高】
- 多くは**構造化データ**(品番→仕様/BOM)。LLMは**自然言語インターフェース**として有効(「品番Xの仕様は?」)。LLMを**真実の源にしない**(出典/原データへリンク)。
- アプローチ: 構造化データをテキスト化して索引、または構造化ストアへのクエリ生成。

### 3-c. 品質データ(測定値・SPC・不良率)  【実現性: 数値計算はLLM不可 → 道具で計算】
- **LLMは生の数値分析(傾向・Cpk・統計)が苦手 → ハルシネーションの温床。絶対に生成させない。**
- 現実解: **計算はコード/集計(SQL・スクリプト)で確定**させ、LLMは**結果を自然言語で説明・要約**。あるいは「先月の不良トップ3」を**クエリ生成**で取り出す。
- ここは皆さんの **Code エージェントが活きる**:品質CSVに対し分析スクリプトを書いて実行(**結果は人がレビュー**)。LLMが数字を“推定”しない設計にする。

**まとめ(テーマ3)**: 図面=メタ検索、製品データ=自然言語IF、品質データ=**計算は道具・説明はLLM**。

## 横断の鉄則(安全・品質)
- 出典必須/資料内のみ回答/「わからない」と言わせる(実装済みの厳格RAG・引用を全面活用)。
- **数値・寸法・統計はLLMに生成させない**(計算は道具、LLMは説明)。
- 品質・検査・安全に関わる出力は**人の確認を必須**化し、監査証跡を残す。
- 文書/データの鮮度管理(更新→索引再構築)のオーナーシップを決める。

## 段階PoC(おすすめ順)
1. **テーマ1(設備トラブル対応)**で成功体験 — 既存RAGで最短、効果が見えやすい。
2. **テーマ2(帳票読み取り)**を1帳票種で — OCR精度を実測し人手確認フローを固める。
3. **テーマ3**は 3-b(製品データ findability)→ 3-c(品質は Code で計算+説明)→ 3-a(図面メタ)の順。

## 実データ受領待ち(実装前に必ず実施 — 規定と同じ原則)
各テーマとも、**実物のサンプルでチャンク/抽出出力を実測**してから実装する。受領時に確認:
- テーマ1: トラブル履歴の記録形式(列・自由記述・写真)
- テーマ2: 実帳票での OCR/抽出 正答率(手書きのクセ)
- テーマ3: 図面タイトルブロックの体裁、品質データの形式(CSV列・粒度)

> 機微情報の扱い: 検証は**ローカルのみ**。実データ本文・図面は**リポジトリに非コミット**。テストは匿名化したダミー最小例に限る。
