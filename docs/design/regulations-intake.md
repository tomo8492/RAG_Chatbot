# 実規定 md の提供ガイド(受け渡し手順)

社内規定の実物(md 等)を提供してもらい、[phase1-regulations.md](phase1-regulations.md) の
設計を**実データで裏取り**してから実装に入るための手順。横断方針(ロードマップ §2)を厳守する。

---

## ⚠ 最重要:安全な受け渡し方法

この実行環境は**毎回リポジトリをクローンし直す使い捨てコンテナ**で、私(AI)が見られるのは
「**ブランチにコミットされたファイル**」か「**チャットに貼付/添付された内容**」だけ。
一方で**実規定の本文はリポジトリにコミットしない**(機微情報)。この2つを両立するため、
次のいずれかで渡してください。

| 方法 | 内容 | 使いどころ |
|---|---|---|
| **A. 自分で実測して結果だけ共有(推奨)** | 実物は手元に置き、下の `inspect_chunks` を**自分のPCで実行**。出力(見出しパスとチャンクの構造)を貼ってもらう | 本文を出したくない/量が多い |
| **B. 匿名化して提供** | 会社名・氏名・金額・固有名詞をダミー化した md にして渡す(コミット可 or 貼付) | 構造はそのまま見たいが中身は伏せたい |
| **C. 一回限りの貼付** | 実本文をチャットに貼付(コミットしない)。私は**匿名化した最小テスト例**と所見だけを成果物にする | 少数・短い規定 |

- いずれも置き場の既定は `data/regulations_intake/`(**`data/` は .gitignore 済み**=誤コミットしない)。
- B でコミットしたい場合のみ、`data/` ではなく committable な場所に置く(例 `examples/`)。**必ず匿名化**。

---

## 1. md の作り方(検証に効く形)

私が見たいのは**構造**(章/条/項/号・附則・別表)が正しく取れるか。完璧な整形は不要。

- 元の**見出しの行**(`第3章 給与` `第12条(基本給)` `附則` `別表第1` など)はできるだけ原文のまま残す。
- 段組み・ヘッダ/フッタ・ページ番号などの**ノイズは混ざってOK**(どう混ざるかも検証対象)。
- 複数規定があれば**1規定=1ファイル**で(就業規則・各課規定など数本あると癖が見える)。
- スキャンPDF(画像)しかない場合はそのままで可(OCR要否も判断する)。

## 2. 確認したいポイント(phase1 §0 のチェックリスト)

提供後、私(または方法Aなら皆さん)が `inspect_chunks` の出力で次を確認する:

- [ ] 章/節/款/条/項/号 の見出し**検出漏れ・誤検出**
- [ ] 漢数字(第十二条)・全角(第１２条)・**枝番(第12条の2)**
- [ ] **附則 / 別表(第N)/ 様式 / 別記 / 別紙 が独立チャンク**になっているか(§0 の既知問題)
- [ ] 定義条(用語の定義)の見出し判定
- [ ] 参照(第N条 / 別表第N / 前条 / 次条 / 前項)が本文に出現し抽出可能か
- [ ] 1条が複数チャンクに割れる頻度・境界(意味の途中で切れていないか)
- [ ] フォーマットの癖: ヘッダ/フッタ・ページ番号・改ページの混入、縦書きPDF、表記ゆれ、スキャンPDF(要OCR)

## 3. 自分で実測する手順(方法 A)

リポジトリ直下で:

```bash
# 1ファイル(先頭200字ずつ表示)
python -m evalkit.inspect_chunks data/regulations_intake/就業規則.md

# フォルダ一括 / 本文を全文表示したいとき
python -m evalkit.inspect_chunks data/regulations_intake/ --full
```

出力は `PATH:`(見出しパス)と `BODY:`(チャンク本文)の一覧＋サマリ。
末尾に **「附則/別表が条に飲み込まれている疑い」** を自動警告する(§0 の裏取り)。
この出力(または気になった箇所)を共有してください。**本文を出したくなければ PATH 行だけでも可**。

## 4. 受領後に私がやること

1. `inspect_chunks` の結果で §2 チェックリストを埋める。
2. 崩れがあれば設計を修正(`splitter._heading` の正規表現・`refs.parse_article`)し、
   phase1-regulations.md の §0 と該当章に追記。
3. **匿名化した最小例**だけを `tests/`(`test_splitter`/`test_refs`)に追加。
4. 裏取り完了 → 承認後にロードマップ **フェーズ0** から実装着手。

> 機微情報は最後までローカル処理。実規定本文・社名・氏名・金額はコミットせず、
> テストもダミーの最小例に限る(ロードマップ §2 横断方針)。
